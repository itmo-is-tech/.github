# Лабораторная работа 5

## Предисловие 🚀

К данному этапу информационная система автосалона уже реализует:

- доменную модель и бизнес-логику автосалона (ЛР1);
- REST API, работу с базой данных и инфраструктурные компоненты (ЛР2);
- аутентификацию и авторизацию через Spring Security и Keycloak (ЛР3);
- микросервисную архитектуру и асинхронное взаимодействие через брокер сообщений (ЛР4).

В рамках лабораторной работы №5 в систему добавляется синхронное межсервисное взаимодействие по gRPC для получения актуального списка автомобилей в наличии.

---

## Цель лабораторной работы

- освоить основы межсервисного взаимодействия через gRPC;
- реализовать gRPC server в `StorageService`;
- реализовать gRPC client в `OrderService`;
- встроить gRPC-вызов в существующие бизнес-сценарии без нарушения правил ЛР1–ЛР4;
- обеспечить базовую отказоустойчивость gRPC-интеграции.

---

## Задание

Необходимо:

- реализовать `gRPC server` в `StorageService`, возвращающий список автомобилей в наличии;
- реализовать `gRPC client` в `OrderService`, запрашивающий этот список;
- определить необходимый набор REST-ручек в `OrderService` для пользовательского сценария;
- сохранить ролевую модель доступа из ЛР3;
- покрыть новую функциональность интеграционными тестами.

---

## Область рассмотрения лабораторной работы

В данной лабораторной работе:

- `StorageService` становится поставщиком данных о складских остатках автомобилей по gRPC;
- `OrderService` становится потребителем этих данных и предоставляет клиентский доступ через REST API;
- бизнес-логика заказов, конфигуратора и событийного взаимодействия (ЛР1–ЛР4) не изменяется.

---

## Архитектурные требования

### StorageService

- поднять gRPC server;
- реализовать gRPC метод получения автомобилей в наличии;
- источник данных - собственная БД `StorageService`;
- возвращать только доступные к продаже автомобили.

### OrderService

- поднять gRPC client;
- выполнять запрос списка автомобилей у `StorageService`;
- обрабатывать таймауты и недоступность `StorageService` корректными ошибками.

### Общие требования

- контракт описать в `.proto` файле;
- логировать межсервисные вызовы для диагностики.

---

## REST API в OrderService

Для данного процесса в `OrderService` необходимо поддержать два REST endpoint, которые получают данные из `StorageService` по gRPC:

- `GET /api/v1/cars` - получить список всех автомобилей в наличии;
- `GET /api/v1/cars/{id}` - получить конкретный автомобиль в наличии по идентификатору.

Оба endpoint должны работать через `gRPC client` обращаться к `StorageService`.

---

## Требования безопасности

- все новые REST-эндпоинты в `OrderService` должны быть защищены;
- доступ:
  - `USER`, `MANAGER`, `ADMIN` - разрешен просмотр доступных автомобилей;
- требования безопасности, реализованные в ЛР3, должны быть сохранены.

---

## Надежность и отказоустойчивость

- настроить timeout для gRPC client;
- при недоступности `StorageService` возвращать контролируемую ошибку (`503 Service Unavailable`).

---

## Тестирование

### Обязательные тесты

- интеграционные тесты gRPC server (`StorageService`);
- интеграционные тесты gRPC client (`OrderService`);
- тесты сценариев ошибок (timeout, service unavailable, пустой результат).

---

## Нефункциональные требования

- совместимость с архитектурой ЛР4;
- сохранение бизнес-правил ЛР1;
- сохранение безопасности ЛР3;
- локальный запуск через `docker-compose`;

---

## Результат выполнения лабораторной работы

В результате выполнения лабораторной работы система должна:

- предоставлять список автомобилей в наличии через REST API `OrderService`;
- получать данные о наличии из `StorageService` по gRPC;
- корректно обрабатывать ошибки межсервисного вызова;
- сохранять целостность существующей бизнес-архитектуры и ролевой модели.
