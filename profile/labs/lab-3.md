# Лабораторная работа 3

## Предисловие 🫡
К данному этапу информационная система автосалона уже реализует:
- доменную модель и бизнес-логику автосалона (ЛР1);
- REST API, работу с базой данных и инфраструктурные компоненты (ЛР2).

В рамках лабораторной работы №3 система продолжает развиваться,
и в неё добавляются механизмы **аутентификации и авторизации**.
Безопасность и разграничение прав рассматриваются **на примере процесса заказа автомобиля**,
однако требования безопасности должны быть применены **ко всем REST-эндпоинтам системы**.


>Бизнес-логика заказа, его жизненный цикл и допустимые переходы между статусами
**определены в лабораторной работе №1** и **не изменяются** в рамках данной лабораторной работы.

---

## Цель лабораторной работы
- освоить основы Spring Security;
- познакомиться с системой централизованной аутентификации и авторизации Keycloak;
- реализовать ролевую модель доступа;
- научиться ограничивать доступ к REST API;
- применить проверки владельца сущностей на уровне методов.

---

## Задание
Необходимо:
- интегрировать Spring Security в существующее приложение;
- настроить Keycloak и модель ролей;
- реализовать авторизацию запросов к REST API;
- ограничить доступ к операциям системы в соответствии с ролями пользователей.

---

## Область рассмотрения лабораторной работы
В лабораторной работе №1 были описаны следующие роли и сценарии:
- клиент;
- менеджер автосалона;
- администратор склада;
- администратор системы;
- управление автомобилями, запчастями, тест-драйвами и заказами.

В рамках лабораторной работы №3:
- механизмы безопасности **обязательны для всех REST-эндпоинтов системы**;
- детальная проработка ролевой модели рассматривается
  на примере операций с заказом автомобиля;
- остальные эндпоинты системы также должны быть защищены,
  но без детального описания сценариев.

---

## Пользователи и роли безопасности

В системе должны быть реализованы следующие **роли безопасности**:

- **USER** — клиент автосалона:
    - может создавать заказы;
    - может просматривать **только свои** заказы;
    - может отменять **только свои** заказы;
    - имеет доступ только к разрешённым пользовательским операциям системы.

- **MANAGER** — менеджер автосалона:
    - имеет доступ ко всем заказам клиентов;
    - выполняет служебные операции над заказами
      в соответствии с бизнес-логикой, определённой в ЛР1;
    - при поиске и просмотре заказов видит **все заказы**.

- **WAREHOUSE_ADMIN** — администратор склада:
    - выполняет операции согласования заказов со складом;
    - имеет доступ к операциям складского уровня.

- **ADMIN** — администратор системы:
    - имеет полный доступ ко всем REST-эндпоинтам системы;
    - может выполнять любые операции без ограничений.

> Один пользователь может обладать **несколькими ролями одновременно**
> (например, менеджер может также иметь роль USER).

---

## Аутентификация и авторизация

- все пользователи обязаны проходить аутентификацию;
- каждый HTTP-запрос проходит авторизацию;
- проверка прав осуществляется на основе ролей пользователя;
- все механизмы аутентификации и авторизации реализуются **через Keycloak**.

Идентификатор текущего пользователя должен извлекаться
из **Security Context**, а не передаваться с клиента.

---

## Работа с заказом

### Общие положения
В данной лабораторной работе используется сущность **«Заказ»**,
реализованная в лабораторной работе №1 и сохранённая в БД в лабораторной работе №2.

- состав сущности;
- статусы заказа;
- жизненный цикл и правила переходов

**не подлежат изменению**.

Цель лабораторной работы — обеспечить,
чтобы каждая операция над заказом
могла быть выполнена **только пользователем,
имеющим соответствующие права доступа**.

---

## Поиск и просмотр заказов

- При выполнении поиска заказов:
    - пользователь с ролью **USER** видит **только свои заказы**;
    - пользователь с ролью **MANAGER** видит **все заказы**;
    - пользователь с ролью **ADMIN** видит **все заказы**.
- Ограничение должно быть реализовано:
    - на уровне сервисного слоя;
    - и/или на уровне запросов к базе данных.

---

## Операции с заказом и права доступа

| Операция                           | Роль                               |
|------------------------------------|------------------------------------|
| Создание заказа                    | USER                               |
| Отмена заказа                      | USER (только владелец заказа)      |
| Операции согласования со складом   | WAREHOUSE_ADMIN                    |
| Служебные операции над заказом     | MANAGER                            |
| Получение списка заказов           | USER (только свои), MANAGER, ADMIN |
| Получение заказа по идентификатору | USER (только свой), MANAGER, ADMIN |
| Любые операции системы             | ADMIN                              |

Для операций, доступных только владельцу заказа,
необходимо реализовать проверку принадлежности заказа пользователю.

---

## Требование к @PreAuthorize (OwnerId)

Для операций, в которых доступ разрешён
только владельцу заказа, необходимо:

- использовать аннотации `@PreAuthorize`;
- проверять, что идентификатор владельца заказа
  совпадает с идентификатором текущего пользователя
  из Security Context.

Пример требований:
- USER не может получить или отменить заказ,
  созданный другим пользователем;
- MANAGER и ADMIN не ограничены проверкой владельца.

---

## Архитектурные требования безопасности

- все REST-эндпоинты системы должны быть защищены;
- ограничение доступа должно быть реализовано:
    - на уровне REST API;
    - на уровне сервисного слоя;
- использовать декларативные механизмы Spring Security;
- запрещено реализовывать собственную систему пользователей.

---

## Задачи лабораторной работы
- подключить Spring Security к проекту;
- настроить Keycloak:
    - создать необходимые роли;
    - создать пользователей;
    - настроить клиент для приложения;
- реализовать интеграцию Spring Security и Keycloak;
- реализовать роль администратора системы;
- защитить все REST-эндпоинты приложения;
- реализовать проверки владельца заказа с использованием `@PreAuthorize`;
- ограничить доступ к поиску и просмотру заказов.

---

## Нефункциональные требования
- аутентификация и авторизация реализуются исключительно через Keycloak;
- keycloak должен подниматься в Docker-контейнере;
- использовать Spring Security без кастомной реализации пользователей.
